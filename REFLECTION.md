# Project Reflection

### 1. Hardest bug and debugging process
The most challenging technical hurdle was a silent failure in the Supabase integration mid-development. After switching from a local prototype to a production database, the application started returning 404 errors on report generation. My initial debugging focused on RLS (Row Level Security) policies, assuming the database was rejecting the inserts. However, after inspecting the network logs, I realized the requests were being sent to `/rest/v1/rest/v1/reports`. 

The issue stemmed from a manual fetch implementation where I had accidentally hardcoded the API prefix while the base URL environment variable already contained it. To resolve this properly and prevent future regressions, I stripped the manual fetch calls and implemented the official `@supabase/supabase-js` SDK. This provided better type safety and handled URL construction internally. A second significant bug involved handling OpenAI 429 rate limits. I had to build a robust `try/catch` wrapper around the AI summary generation that seamlessly falls back to a deterministic, programmatic summary. Testing this required intentionally lowering my API quota to trigger the failure state and ensure the UI didn't break for the end-user.

### 2. Decision reversed mid-week
Initially, I planned for the entire audit report to be generated by an LLM based on the user's input. Mid-week, I realized this was a mistake. Relying on AI for the core financial calculations led to "hallucinated" waste estimates—sometimes the AI would miss a simple seat-count overlap or miscalculate the cost of a Pro tier. 

I reversed this decision and moved to a "Deterministic-First" architecture. I built a standalone rules engine in `/lib/engine` that uses hardcoded, unit-tested logic for cost calculations and overlap detection. This ensured the numbers were always 100% accurate. I limited the AI's role to generating the "Executive Summary" section, providing a human-readable narrative over the hard data. This tradeoff drastically improved the product's credibility, as users are much more forgiving of a slightly generic summary than an incorrect dollar amount on their waste report.

### 3. What would be built in week 2
In the second week, I would focus on "Continuous Audit" and deeper integrations. Currently, StackAudit is a point-in-time assessment. I would build a Slack integration that alerts founders whenever a new recurring charge is detected in a connected accounting system like Quickbooks or Mercury. 

Technically, this would require shifting from manual user input to OAuth-based financial data ingestion. I would also implement "Multi-Organization" support, allowing fractional CFOs and agency owners to manage audits for multiple clients under a single dashboard. Finally, I’d build a "Savings Verification" feature, where users can upload proof of a cancelled subscription to update their global "Total Savings" leaderboard, gamifying the optimization process for the team.

### 4. How AI tools were used + where AI failed
I used AI primarily for scaffolding the initial Tailwind layouts and generating the deterministic rule templates. It was incredibly efficient at creating the multi-step audit form components and the initial scoring logic.

However, AI failed significantly when it came to complex Radix UI accessibility requirements. It generated `Dialog` and `Sheet` components that lacked proper `DialogTitle` children, leading to console warnings and broken screen-reader support. I had to manually go back and inject `VisuallyHidden` wrappers into the core UI library. AI also struggled with the nuances of the Supabase SDK's error handling, often suggesting deprecated syntax that led to runtime crashes. Every AI-generated function in the audit engine was manually reviewed and rewritten to ensure the math was strictly deterministic rather than probabilistic.
